# Identity and Access Management (IAM) in Google Cloud

## 1. Core Concept

**Identity and Access Management (IAM)** – A **security framework that controls who (identity) can do what (permissions) on which resource** in Google Cloud.

IAM ensures **secure, controlled access to resources** such as datasets, pipelines, storage, and compute systems.

### Key Goal

Apply the **principle of least privilege** – give users **only the minimum permissions required**.

---

# Key Components of IAM

## 1. Identity

**Identity** – An **entity that can be authenticated** in Google Cloud.

Examples:

* **Google Account** – Individual user account
* **Service Account** – Identity used by **applications or services**
* **Google Group** – Collection of users
* **Cloud Identity / Workspace user**

Example:
A **Dataflow pipeline** running with a **service account** to read data from Cloud Storage and write to BigQuery.

---

## 2. Resource

**Resource** – Any **Google Cloud object** that access can be granted to.

Examples:

* Project
* BigQuery dataset
* Cloud Storage bucket
* Pub/Sub topic
* Compute Engine VM

Example hierarchy:

```
Organization
   │
   ├── Folder
   │      ├── Project
   │             ├── BigQuery Dataset
   │             ├── Storage Bucket
```

Permissions **inherit down the hierarchy**.

---

## 3. Role

**Role** – A **collection of permissions** that defines allowed actions.

Types of roles:

### 1️⃣ Basic Roles

Legacy roles applied at project level:

* **Viewer** – Read-only access
* **Editor** – Read + modify
* **Owner** – Full control

⚠️ **Exam Tip:** Avoid basic roles in production due to excessive permissions.

---

### 2️⃣ Predefined Roles

**Predefined roles** – Google-managed roles designed for specific services.

Examples:

* **BigQuery Data Viewer**
* **BigQuery Data Editor**
* **Storage Object Viewer**

Example:

```
roles/bigquery.dataViewer
```

Allows:

* Read tables
* Query data

---

### 3️⃣ Custom Roles

**Custom roles** – Roles **created by an organization with specific permissions**.

Use case:
Grant only:

```
bigquery.tables.get
bigquery.tables.list
```

---

## 4. Policy

**IAM Policy** – A **binding between identity and role on a resource**.

Structure:

```
Member → Role → Resource
```

Example:

```
User: analyst@company.com
Role: BigQuery Data Viewer
Resource: Project analytics-prod
```

---

# Organization Policies

**Organization Policy** – A **central governance mechanism that enforces rules across an organization**.

Used for **compliance, security, and governance**.

Examples:

| Policy                               | Purpose                  |
| ------------------------------------ | ------------------------ |
| Restrict resource locations          | Enforce data residency   |
| Disable service account key creation | Prevent credential leaks |
| Require CMEK encryption              | Enforce encryption       |
| Restrict external IPs                | Security hardening       |

Example:

```
constraints/gcp.resourceLocations
```

Restricts resources to:

```
europe-west1
europe-west4
```

# IAM in Data Engineering Architectures

Data engineers must control **who can access data pipelines and datasets**.

Example architecture:

```
Pub/Sub → Dataflow → BigQuery → Looker
```

IAM roles:

| Service  | Role                 |
| -------- | -------------------- |
| Pub/Sub  | Pub/Sub Subscriber   |
| Dataflow | Dataflow Worker      |
| BigQuery | BigQuery Data Editor |
| Looker   | BigQuery Data Viewer |

---

# Service Accounts (Critical Exam Topic)

**Service Account** – A **special Google account used by applications instead of humans**.

Example:

```
dataflow-worker@project.iam.gserviceaccount.com
```

Use cases:

* Dataflow writing to BigQuery
* Cloud Functions accessing Pub/Sub
* Dataproc accessing Cloud Storage

Best practice:

✔ Use **Workload Identity**
✔ Avoid **service account keys**

---

# IAM Best Practices

### 1️⃣ Principle of Least Privilege

Grant **minimum permissions required**.

Bad:

```
roles/editor
```

Good:

```
roles/bigquery.dataViewer
```

---

### 2️⃣ Use Groups Instead of Individual Users

Instead of:

```
user1@company.com
user2@company.com
user3@company.com
```

Use:

```
data-analysts@company.com
```

---

### 3️⃣ Separate Environments

Use different projects:

```
dev-project
test-project
prod-project
```

This prevents accidental production access.

---

### 4️⃣ Use IAM Conditions

**IAM Conditions** – Allow access **only under certain conditions**.

Example:

```
Allow access only from corporate IP range
Allow access only during working hours
```

# Common Exam Traps ⚠️

### Trap 1

Using **Owner/Editor roles**.

✔ Correct answer: **Use predefined roles**.

---

### Trap 2

Granting **permissions to individual users**.

✔ Correct answer: **Grant permissions to groups**.

---

### Trap 3

Using **service account keys**.

✔ Correct answer: **Use Workload Identity or IAM roles**.

---

### Trap 4

Granting permissions at **project level instead of dataset level**.

✔ Best practice: **grant lowest possible resource level**.

---

# Quick Revision Summary

| Concept             | Meaning                                 |
| ------------------- | --------------------------------------- |
| Identity            | User, group, or service account         |
| Resource            | Cloud object (project, dataset, bucket) |
| Role                | Collection of permissions               |
| Policy              | Identity + Role + Resource              |
| Service Account     | Identity for applications               |
| Organization Policy | Governance constraints across org       |

**Golden Rule for the Exam**

```
Identity + Role + Resource = IAM Access
```
