# **Topic: 4. Regional considerations (data sovereignty) for data access and storage**

Data sovereignty refers to **ensuring that data is stored, processed, and accessed according to the laws and regulations of the country or region where the data resides**. In Google Cloud architecture, this means **carefully selecting regions, controlling cross-border data movement, and applying governance policies**.

---

# 1. Data Sovereignty

**Data Sovereignty — the concept that data is subject to the laws and governance of the country where it is stored.**

Example:

* European citizen data stored in the EU must comply with **GDPR**.

This means organizations must control:

* Where data is stored
* Where data is processed
* Who can access the data

Exam Tip:
If the exam says **“data must remain in a specific country or region” → think regional storage configuration.**

---

# 2. Google Cloud Regions and Locations

Google Cloud allows selecting **specific regions** for data storage and processing.

**Region — a geographic location containing multiple availability zones where cloud resources are deployed.**

Example:

* `europe-west1` – Belgium
* `us-central1` – Iowa
* `asia-east1` – Taiwan

Example service:

**BigQuery**

When creating datasets, you must choose a **location**.

Example:

* EU companies often select **EU region** to comply with GDPR.

---

# 3. Multi-Region Storage

**Multi-region — a storage configuration where data is replicated across multiple regions within a geographic area.**

Example:

* **EU multi-region**
* **US multi-region**

Example service:

**Cloud Storage**

Advantages:

* Higher availability
* Disaster recovery

Trade-off:

* Data may move between regions **within the multi-region boundary**.

Exam Tip:
If data **must stay within the EU**, using **EU multi-region** is acceptable.

---

# 4. Restricting Data Location Using Organization Policies

Organizations can enforce **location restrictions**.

Service:

**Cloud Resource Manager**

Policy example:

**Location constraint**

* Prevent resources from being created outside approved regions.

Example:

```
Allowed regions: europe-west1, europe-west4
```

Benefit:
Ensures developers cannot accidentally store data in other countries.

---

# 5. Regional Processing in Data Pipelines

Data processing services should also run in the **same region as the data**.

Example pipeline:

1. Data ingestion through **Pub/Sub**
2. Processing via **Dataflow**
3. Storage in **BigQuery**

Best practice:
Run **Pub/Sub, Dataflow, and BigQuery in the same region**.

Benefits:

* Reduced latency
* Avoids cross-region data transfer
* Maintains compliance

---

# 6. Cross-Border Data Transfer Risks

Moving data between regions may:

* Violate regulations
* Increase latency
* Increase network costs

Example:
EU customer data processed in US servers may violate **GDPR data transfer rules**.

Therefore:

✔ Keep **storage and processing in the same region**.

---

# 7. Example Architecture (EU Compliance)

Scenario:
A European company must ensure customer data never leaves the EU.

Architecture:

1. Data stored in **BigQuery EU region**
2. Files stored in **Cloud Storage EU multi-region**
3. Processing with **Dataflow EU region**
4. Access controlled via **Cloud IAM**

Result:

* Data stays inside EU
* GDPR compliance maintained

---

# 8. Exam Tips

Common exam patterns:

If the question mentions:

| Requirement                      | Correct Approach                 |
| -------------------------------- | -------------------------------- |
| Data must stay in EU             | Use EU region or EU multi-region |
| Prevent data from leaving region | Use organization location policy |
| Reduce latency                   | Process data in same region      |
| Regulatory compliance            | Choose region carefully          |

Common exam trap:

❌ Storing data in one region and processing in another
✔ Always keep **data + compute in the same region** when possible.

---

# Quick Revision Summary

**Data Sovereignty** – data must follow the laws of the country where it resides.
**Region** – geographic location where cloud resources run.
**Multi-region** – data replicated across multiple regions in one geography.
**Location constraints** – policies restricting where resources can be deployed.
**Best practice** – keep **data storage and processing in the same region**.

---

✅ **1-Sentence Exam Memory Trick**

**Choose the correct region → restrict resource locations → keep processing and storage in the same geography.**

# Google Cloud Location Types (Region vs Multi-Region vs Dual-Region vs Zone)

| Location Type    | Definition                                                         | Availability | Typical Use Case            |
| ---------------- | ------------------------------------------------------------------ | ------------ | --------------------------- |
| **Zone**         | Single deployment area within a region                             | Lowest       | Compute workloads           |
| **Region**       | Geographic location with multiple zones                            | High         | Data processing and storage |
| **Dual-Region**  | Data replicated across two specific regions                        | Very High    | Disaster recovery           |
| **Multi-Region** | Data replicated across multiple regions in a large geographic area | Highest      | Global access               |

---

# 1. Zone

**Zone — a deployment area within a region containing compute, storage, and networking resources.**

Example:

* `us-central1-a`
* `europe-west1-b`

Each region contains **multiple zones**.

Example services running in zones:

* **Compute Engine**
* **Google Kubernetes Engine**

Example:
A VM instance running in **us-central1-a**.

Exam Tip:
Zones are mainly used for **compute resources**, not data residency decisions.

---

# 2. Region

**Region — a geographic location containing multiple zones where services are deployed.**

Example regions:

* `us-central1` (Iowa)
* `europe-west1` (Belgium)
* `asia-east1` (Taiwan)

Example services using regions:

* **BigQuery**
* **Cloud SQL**
* **Dataflow**

Advantages:

* Data stays within a specific geography
* Lower latency within the region
* Better compliance with **data sovereignty laws**

Exam Tip:
If a question says **“data must remain in a specific country or region” → choose a regional deployment**.

---

# 3. Dual-Region

**Dual-Region — a storage configuration where data is replicated across two specific regions.**

Example:
`europe-west1 + europe-west4`

Supported mainly in:

* **Cloud Storage**

Benefits:

* High availability
* Disaster recovery
* Predictable replication locations

Example use case:
A financial system requires **high availability but must keep data inside Europe**.

Exam Tip:
If the question mentions:

* **high availability**
* **controlled replication**
* **two specific regions**

→ choose **dual-region storage**.

---

# 4. Multi-Region

**Multi-Region — data replicated across multiple regions within a large geographic area.**

Examples:

* **US**
* **EU**
* **ASIA**

Example services:

* **BigQuery multi-region datasets**
* **Cloud Storage multi-region buckets**

Advantages:

* Highest availability
* Global analytics performance
* Automatic replication

Example:
A global analytics platform stores data in **BigQuery EU multi-region**.

Exam Tip:
Multi-region is commonly used for **analytics and large-scale data warehousing**.

---

# Comparison Diagram (Conceptual)

```
Zone
   ↓
Region (multiple zones)
   ↓
Dual-Region (two regions)
   ↓
Multi-Region (multiple regions)
```

Availability increases as you move down the list.
