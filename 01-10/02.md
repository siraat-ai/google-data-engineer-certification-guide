# **Topic: 2. Data security (encryption and key management)**

Data security ensures that **sensitive data is protected from unauthorized access, leaks, or tampering** during **storage, processing, and transmission**. In Google Cloud data engineering architectures, this is achieved primarily through **encryption** and **secure key management**.

---

# 1. Encryption Basics

**Encryption — transforming readable data into an unreadable format using cryptographic algorithms so only authorized entities can decrypt it.**

### Types of Encryption

**Encryption at Rest — data is encrypted when stored on disk or storage systems.**
**Encryption in Transit — data is encrypted while moving across networks using secure protocols.**

### Google Cloud Implementation

* **Cloud Storage** – automatically encrypts stored objects.
* **BigQuery** – encrypts all stored datasets and tables by default.
* **Cloud SQL** – encrypts database storage and backups.

### Encryption in Transit

Uses secure protocols such as:

* **TLS (Transport Layer Security) — protocol that encrypts communication between systems over networks.**

Example:

* Data flowing from **Pub/Sub** to **Dataflow** is encrypted using TLS.

---

# 2. Default Encryption in Google Cloud

Google Cloud automatically encrypts all data using **AES-256 encryption**.

**AES-256 — Advanced Encryption Standard using a 256-bit key, considered highly secure.**

Default encryption includes:

* Storage systems
* Snapshots
* Temporary files
* Backups

Exam Tip:
✅ **Google Cloud encrypts all customer data by default without user action.**

---

# 3. Key Management

**Encryption keys — cryptographic values used to encrypt and decrypt data.**

Key management defines **how keys are created, stored, rotated, and controlled**.

Primary service:

**Cloud Key Management Service**

Capabilities:

* Create and manage encryption keys
* Rotate keys automatically
* Control access to keys
* Audit key usage

Example:
A company storing sensitive customer data in **BigQuery** may encrypt datasets using **customer-managed encryption keys (CMEK)** stored in Cloud KMS.

---

# 4. Encryption Key Options in Google Cloud

### 1. Google-Managed Encryption Keys (GMEK)

**Google-managed encryption keys — encryption keys fully managed by Google without customer control.**

Characteristics:

* Default option
* Automatic rotation
* No operational overhead

Example:
Data stored in **BigQuery** is encrypted with Google-managed keys by default.

---

### 2. Customer-Managed Encryption Keys (CMEK)

**Customer-managed encryption keys — keys created and controlled by customers in Cloud KMS.**

Advantages:

* Control key lifecycle
* Key rotation policies
* Ability to disable or revoke keys

Example:
A financial company stores transaction data in **Cloud Storage** but manages encryption keys through **Cloud KMS**.

Exam Tip:
If the question says **“customer must control encryption keys” → choose CMEK**.

---

### 3. Customer-Supplied Encryption Keys (CSEK)

**Customer-supplied encryption keys — encryption keys provided by the customer during API requests and not stored by Google.**

Characteristics:

* Maximum control
* Higher operational complexity
* Google does not store the key

Used when organizations have strict compliance requirements.

---

# 5. Hardware Security for Keys

**Cloud HSM**

**Hardware Security Module (HSM) — a physical device that securely stores cryptographic keys and performs encryption operations.**

Used when regulations require **hardware-based key protection**.

Example:
Banks may store encryption keys in **Cloud HSM** to meet financial compliance standards.

---

# 6. Identity and Access Control for Keys

Key access is controlled through:

**Cloud IAM**

**IAM — system for defining who can access resources and what actions they can perform.**

Example roles:

* **Cloud KMS Admin** — manage keys
* **Cloud KMS CryptoKey Encrypter/Decrypter** — encrypt/decrypt data

Best Practice:
Use **least privilege principle** when granting key access.

---

# 7. Practical Architecture Example

A secure analytics pipeline:

1. Data ingested via **Pub/Sub**
2. Processed with **Dataflow**
3. Stored in **BigQuery**
4. Data encrypted using **CMEK in Cloud KMS**
5. IAM controls who can decrypt the data

Security benefits:

* Data protected in transit
* Data encrypted at rest
* Customer controls encryption keys

---

# 8. Exam Tips

Important exam patterns:

* **Google encrypts data at rest by default**
* **TLS is used for encryption in transit**
* **CMEK = customer controls keys**
* **Cloud KMS = centralized key management**
* **Cloud HSM = hardware-backed key storage**
* **IAM controls key access**

Common exam trap:
❌ Thinking encryption must be manually enabled
✔ It is **enabled by default in Google Cloud**

**Encryption** – protects data by converting it into unreadable format.

**GMEK** – keys managed by Google.
**CMEK** – keys managed by customer.
**CSEK** – keys supplied by customer per request.

# 1. Google-Managed Encryption Keys (GMEK)

**Google-Managed Encryption Keys — encryption keys automatically created and managed by Google Cloud.**

Characteristics:

* Default encryption for most Google Cloud services
* No setup required
* Automatic key rotation
* Minimal operational overhead

Example services using GMEK by default:

* **BigQuery**
* **Cloud Storage**
* **Cloud SQL**

Exam Tip:
If the scenario **does not require customer control over encryption keys → choose GMEK**.

---

# 2. Customer-Managed Encryption Keys (CMEK)

**Customer-Managed Encryption Keys — encryption keys created and controlled by the customer using Cloud KMS.**

Characteristics:

* Customer controls **key lifecycle**
* Keys can be **disabled, rotated, or deleted**
* Access is controlled via **Cloud IAM**

Example:
A company storing financial data in **BigQuery** wants the ability to **revoke access to encrypted data** by disabling the encryption key in **Cloud KMS**.

Exam Tip:
If the question says:

* **“customer must manage encryption keys”**
* **“security team must control key rotation”**
* **“ability to revoke encryption access”**

✔ The answer is **CMEK**.

---

# 3. Customer-Supplied Encryption Keys (CSEK)

**Customer-Supplied Encryption Keys — encryption keys provided by the customer during API requests and not stored by Google Cloud.**

Characteristics:

* Maximum customer control
* Keys must be supplied **every time data is accessed**
* Google never stores the encryption key

Typical use case:
Highly regulated environments where **cloud providers must not retain encryption keys**.

Limitations:

* Operationally complex
* Limited service support
* Risk if key is lost (data becomes unrecoverable)

Exam Tip:
If the scenario says:

* **Google must not store encryption keys**
* **customer provides encryption keys with each request**

✔ The answer is **CSEK**.

---

# Quick Exam Decision Rule

Use this **simple memory rule**:

| Requirement in Question         | Correct Key Type |
| ------------------------------- | ---------------- |
| No special security requirement | **GMEK**         |
| Customer must control keys      | **CMEK**         |
| Google must never store keys    | **CSEK**         |
