# **Topic: 3. Privacy (e.g., strategies to handle personally identifiable information)**

Privacy in data engineering focuses on **protecting sensitive personal data and ensuring that personally identifiable information (PII) is handled securely and compliantly** across data pipelines, storage, and analytics systems.

---

# 1. Personally Identifiable Information (PII)

**Personally Identifiable Information (PII) — any data that can identify a specific individual directly or indirectly.**

Examples of PII:

* Full name
* Email address
* Phone number
* National ID or Social Security Number
* Passport number
* Home address
* IP address
* Biometric identifiers

Example:
A dataset containing **customer name + email + address** contains PII.

Exam Tip:
If the exam mentions **user identity or personal data**, it usually refers to **PII protection strategies**.

---

# 2. Privacy Protection Strategies

Data engineers must implement **privacy-preserving techniques** when processing or analyzing data.

---

## 2.1 Data Masking

**Data Masking — replacing sensitive data with fictional or obfuscated values while preserving the format.**

Example:

| Original Data                                       | Masked Data    |
| --------------------------------------------------- | -------------- |
| [john.smith@gmail.com](mailto:john.smith@gmail.com) | j***@gmail.com |
| 123-45-6789                                         | ***-**-6789    |

Use cases:

* Analytics environments
* Development and testing datasets
* BI dashboards

Google Cloud tool:

**Cloud Data Loss Prevention**

Capabilities:

* Detect PII automatically
* Mask or redact sensitive data
* Tokenize data

---

## 2.2 Data Anonymization

**Data Anonymization — permanently removing identifiable information so individuals cannot be identified.**

Techniques:

* Removing identifiers
* Aggregating data
* Generalizing values

Example:

| Original               | Anonymized       |
| ---------------------- | ---------------- |
| John Smith, 34, Madrid | Age group: 30–40 |

Key characteristic:

* Data **cannot be traced back to the individual**.

Use case:
Publishing **public datasets** or research data.

---

## 2.3 Data Pseudonymization

**Pseudonymization — replacing identifying fields with artificial identifiers while keeping the possibility of re-identification using a secure mapping table.**

Example:

| User ID | Name       |
| ------- | ---------- |
| U1001   | John Smith |

Benefits:

* Protects identity
* Allows linking records for analysis

Example implementation:

* Store pseudonymized data in **BigQuery**
* Store the mapping table in a **secure system with restricted access**

---

## 2.4 Tokenization

**Tokenization — replacing sensitive data with randomly generated tokens that map to the original value stored securely.**

Example:

| Original Credit Card | Token    |
| -------------------- | -------- |
| 4532 1234 5678 9012  | TKN98234 |

Used for:

* Payment processing
* Secure data sharing

Tokenization can also be implemented using **Cloud DLP tokenization features**.

---

# 3. Data Minimization

**Data Minimization — collecting and storing only the data that is necessary for business purposes.**

Example:
Instead of storing **full birth date**, store only **age range**.

Benefits:

* Reduced security risk
* Lower compliance burden

Exam Tip:
Many privacy questions expect **“store the minimum amount of data required.”**

---

# 4. Access Control for Sensitive Data

Access to PII must be restricted.

Key service:

**Cloud IAM**

**IAM — system for defining who can access resources and what actions they can perform.**

Example:

* Data analysts can query **aggregated datasets**
* Only compliance team can access raw PII data

---

# 5. Column-Level and Row-Level Security

To protect sensitive fields within datasets.

### Column-Level Security

**Column-level security — restricting access to specific columns in a dataset.**

Example:
In a table:

| Name | Email | Credit Card |
| ---- | ----- | ----------- |

Analysts can access:

* Name
* Email

But **not credit card number**.

Implemented in:

**BigQuery**

---

### Row-Level Security

**Row-level security — restricting access to specific rows based on user attributes.**

Example:
Regional managers can only see **customers from their region**.

---

# 6. Privacy in Data Pipelines

Example architecture for handling PII safely:

1. Data ingested via **Pub/Sub**
2. Processed using **Dataflow**
3. PII detected and masked using **Cloud Data Loss Prevention**
4. Masked data stored in **BigQuery**

Benefits:

* Sensitive fields never exposed to analytics users.

---

# 7. Regulatory Compliance

Privacy strategies help comply with regulations such as:

* **GDPR**
* **HIPAA**
* **CCPA**

Common compliance requirements:

* Protect personal data
* Restrict access
* Enable auditing and tracking

---

# 8. Exam Tips

Key exam patterns:

If question mentions **sensitive personal data → think privacy protection techniques**.

Common correct answers:

* **Cloud DLP for PII detection**
* **Data masking**
* **Tokenization**
* **IAM access control**
* **Column-level security**

Common exam trap:
❌ Storing raw PII in analytics datasets
✔ Mask or anonymize PII before analytics.

# Quick Revision Summary

- **PII** – Data that identifies a person.

- **Data Masking** – Hide sensitive values but keep the same format.

- **Anonymization** – Permanently remove identity from data.

- **Pseudonymization** – Replace identifiers but allow re-identification through a mapping.

- **Tokenization** – Replace sensitive data with tokens stored securely.

- **Cloud DLP** – Detects and masks sensitive information.

- **IAM** – Controls access to sensitive data.

- **Column-Level Security** – Restricts access to sensitive fields.

- **Row-Level Security** – Restricts access to specific records.

Below are the **5 most common Professional Data Engineer exam scenarios involving PII and privacy protection**. These appear frequently in **case-study questions** where you must choose the **best Google Cloud service or architecture decision**.

---

# 1. Automatically Detect and Mask PII in Data Pipelines

### Scenario

A company ingests customer data containing **names, emails, phone numbers, and credit card numbers**. They want to **automatically detect and mask PII before storing it for analytics**.

### Correct Solution

Use:

* **Cloud Data Loss Prevention**
* **Dataflow**

### Architecture

1. Data ingestion through **Pub/Sub**
2. Pipeline processing with **Dataflow**
3. Detect and mask PII using **Cloud DLP**
4. Store sanitized data in **BigQuery**

### Exam Tip

If the question says **“automatically detect PII”**, the answer is almost always **Cloud DLP**.

---

# 2. Analysts Must Query Data Without Seeing Sensitive Fields

### Scenario

Data analysts need to run analytics on customer datasets, but they **must not see sensitive fields like SSN or credit card numbers**.

### Correct Solution

Use **column-level security** in:

* **BigQuery**

### Implementation

Restrict columns such as:

* SSN
* Credit card number
* Personal phone numbers

Only authorized roles can access those columns.

Also control permissions with:

* **Cloud IAM**

### Exam Tip

If question mentions **“restrict access to specific columns” → BigQuery column-level security.**

---

# 3. Protect PII in Development and Test Environments

### Scenario

Developers need access to production-like datasets for testing, but **real customer data cannot be exposed**.

### Correct Solution

Use:

* **Cloud DLP data masking**
* **Tokenization**

Example:

| Production Data                                     | Masked Test Data                              |
| --------------------------------------------------- | --------------------------------------------- |
| [john.smith@gmail.com](mailto:john.smith@gmail.com) | [user123@gmail.com](mailto:user123@gmail.com) |

### Workflow

1. Export production dataset
2. Run **Cloud DLP masking**
3. Store sanitized dataset in **BigQuery test project**

### Exam Tip

If question says **“safe dataset for development or testing” → masking or anonymization**.

---

# 4. Secure Data Sharing with External Partners

### Scenario

A company wants to share data with a partner but **cannot expose personally identifiable information**.

### Correct Solution

Use:

* **Data anonymization**
* **Aggregation**

Example:

Instead of sharing:

| Name       | Purchase Amount |
| ---------- | --------------- |
| John Smith | $200            |

Share:

| Age Group | Average Purchase |
| --------- | ---------------- |
| 30–40     | $180             |

Store and publish the dataset using:

* **BigQuery datasets**
* **Analytics sharing**

Example service:

* **Analytics Hub**

### Exam Tip

If question mentions **external sharing of datasets → anonymize or aggregate data first.**

---

# 5. Regional Privacy Regulations (GDPR)

An EU company must ensure **customer personal data remains protected and handled according to privacy regulations**.
